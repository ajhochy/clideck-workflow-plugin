# CliDeck Workflow Plugin — Design

**Date:** 2026-05-06
**Status:** Design approved, awaiting implementation plan

## Purpose

A CliDeck plugin that turns a plain-language description of work into a finished, smoketested pull request — automatically running planning, GitHub issue creation, sequential implementation, and Codex-driven smoketesting end-to-end. The user is interrupted only for clarifying questions during planning. Failures trigger a bounded self-heal loop. The plugin produces a non-developer summary document at the end.

## Workflow Overview

Four stages, run in order, each as its own CliDeck session under the project:

1. **Plan** (Opus + Haiku subagents)
2. **Issues** (Sonnet)
3. **Pipeline** (Sonnet orchestrator + Sonnet workers, one issue at a time, each in its own worktree)
4. **Smoketest** (Codex)

A failed smoketest spawns a *focused fix sub-workflow* (Stages 1→4 against just the failures), max 2 retries.

## Architectural Approach: One Session Per Stage

Each stage runs in its own CliDeck session, spawned in sequence under the user's selected project. Sessions are clearly named (`Workflow #N · Plan`, `Workflow #N · Pipeline`, etc.). When a stage finishes, the plugin closes its session and spawns the next.

State between stages lives in a per-workflow JSON file. Sessions are ephemeral; the file persists.

## Workflow State

Each workflow gets a folder at:

```
~/.clideck/plugins/<plugin-id>/workflows/<workflow-id>/
  state.json
  done/                  # marker files written by stages
    planning.done
    issues.done
    pipeline.done
    smoketest.done
  logs/                  # per-stage transcripts archived after close
  smoketest.md           # generated by Stage 4, also committed to branch
  summary.md             # generated at end, also committed + used as PR description
```

`state.json` schema (informal):

```jsonc
{
  "id": "wf-2026-05-06-001",
  "title": "Add login throttling",
  "description": "<original user input>",
  "projectId": "...",
  "githubRepo": "owner/repo" | null,
  "branch": "feat/login-throttling",
  "branchSource": "user" | "auto",
  "currentStage": "planning" | "issues" | "pipeline" | "smoketest" | "fix" | "done" | "failed",
  "plan": { /* atomic steps with file refs, deps, success criteria */ },
  "issues": [ { "number": 34, "title": "...", "order": 1, "status": "..." } ],
  "manualSetup": [ { "title": "...", "steps": [...] } ],
  "rhythmTask": { "id": "...", "url": "..." } | null,
  "pr": { "number": 12, "url": "...", "draft": true },
  "smoketestResult": { "status": "passed" | "failed", "failures": [...] },
  "fixAttempts": [ /* nested mini-state per retry */ ],
  "createdAt": "...",
  "updatedAt": "..."
}
```

## Stage Details

### Stage 1 — Plan (Opus)

The Opus session is told to:

1. Read `state.json` for description and project context.
2. Dispatch Haiku subagents to thoroughly explore the codebase (architecture, conventions, relevant files, dependencies).
3. After exploration returns, ask the user clarifying questions informed by the codebase. Q&A surface is dual: the live session and the panel chat box (both pipe into the session).
4. Without further user feedback, design an architecture that aligns most closely with the existing codebase (no architecture-feedback loop).
5. Write a detailed plan as **atomic steps**. Each step contains enough context (file paths, function/symbol names, expected behavior, dependencies on prior steps) that an implementing agent can execute it without re-reading the codebase. Cross-step coherence rules are stated explicitly.
6. Write the plan to `state.json.plan` and create marker `done/planning.done`.

The plugin watches the `done/` folder; when `planning.done` appears it closes the session and advances.

### Stage 2 — Issues (Sonnet)

Reads `state.json.plan`. Creates GitHub issues + milestones reflecting the atomic steps. Suggests implementation order. Writes the issue list (number, title, order, dependencies) to `state.json.issues`. If the project has no GitHub repo, falls back to a local TODO list with synthetic IDs in the same field. Marker `issues.done`.

### Stage 3 — Pipeline (Sonnet orchestrator + Sonnet workers)

The orchestrator session:

1. Reads `state.json.issues`; confirms or refines order.
2. **Resolves the branch.** If the user typed a branch name in the new-workflow form, uses it. If "auto", generates a name from issue scope (e.g., `feat/login-throttling`) and creates it from the project's default branch.
3. **Opens a Draft PR** as soon as the first commit lands. PR description is the live `summary.md` (kept fresh each stage). PR stays Draft through pipeline + smoketest; flips to Ready for Review only after smoketest passes.
4. For each issue, sequentially:
   - Create a git worktree off the workflow branch.
   - Dispatch a Sonnet sub-agent into the worktree with that issue's atomic-step context.
   - Sub-agent implements, commits locally, pushes to the workflow branch.
   - Sub-agent watches CI; if red, fixes and re-pushes (bounded retries).
   - On green CI, closes the GitHub issue.
   - Worktree is removed.
5. After all issues land:
   - Gathers the **manual-setup list** (API keys, env vars, GitHub secrets, third-party config) into `state.json.manualSetup`.
   - Calls the **Rhythm App MCP** to create a task titled *"Manual setup for Workflow #N: <title>"* with a step-by-step checklist body. Task ID/URL is stored in `state.json.rhythmTask`.
6. Writes marker `pipeline.done`.

If the Rhythm MCP is unavailable at plugin init, the panel shows a yellow banner; Stage 3 still finishes and the manual-setup list stays in `state.json` for the user to copy.

### Stage 4 — Smoketest (Codex, gated by global lock)

A global "smoketest lock" serializes Stage 4 across all workflows on the machine, since Codex's clickthrough drives shared OS resources (foreground browser, desktop apps, email client). Earlier stages still run in parallel across workflows.

Codex session:

1. Acquires the lock (queues if held; row in panel shows "Queued for smoketest").
2. Reads `state.json` (description, plan, issues, diff vs base, manualSetup).
3. Generates `smoketest.md` — a real Markdown checklist committed to the branch. Includes cross-app checks where implied by the work (email arrived, mobile↔desktop sync, third-party verification, etc.).
4. Executes the checklist by driving browsers, native desktop apps, email — whatever the items require. Annotates each item with ✅/❌, evidence (screenshot path, log snippet, command output).
5. Writes overall outcome to `state.json.smoketestResult`. Marker `smoketest.done`. Releases lock.

### Failure Loop-Back (Stage 4 → Fix Sub-Workflow)

If `smoketestResult.status === 'failed'`:

- Plugin spawns a fix sub-workflow recorded under `state.json.fixAttempts[]`.
- Fix sub-workflow runs an abbreviated Stage 1 (Opus reads failure list, **skips the full architecture pass**, plans only the fixes as new atomic steps), then Stages 2 → 3 → 4 against only those fix steps, on the same branch.
- If the fix smoketest passes → workflow marked `done`, PR flipped to Ready for Review.
- If it fails again → second fix sub-workflow.
- **Maximum 2 fix attempts.** After that, plugin stops, panel shows cumulative failure list with the last agent's suggested next steps, PR remains Draft, workflow marked `failed`.

### Per-Stage Self-Heal (Separate from Loop-Back)

If a non-smoketest stage fails (e.g., Stage 3 sub-agent can't fix CI after its bounded retries), the plugin re-spawns that stage's session once with the failure context appended to the starter prompt. If it fails again, stop and surface.

## Concurrency Rules

- **Different projects, or same project + different branches:** Workflows run in parallel, fully independent.
- **Same project + same branch:** When the user starts a second workflow targeting an in-flight branch, the new-workflow form prompts: pick a different branch name or wait. (No silent collision.)
- **Smoketest stage:** A global lock serializes Stage 4 across the entire machine.

## UI: Toolbar Panel

A toolbar button toggles a small panel anchored to the button. Two views.

### View 1 — Workflow List (default)

Vertical list of active and recent workflows. Each row:

- Auto-derived title (editable)
- Project name + branch
- Thin progress bar with one-line subtext (e.g., *"Stage 3 of 4 · Pipeline running issue #34"*)
- Status dot: green = running, amber = needs you, red = failed, grey = done

A "+ New Workflow" button at the top.

Clicking a row expands inline:

- Four stages as a checklist with timestamps
- Links to each spawned CliDeck session
- During Planning: a chat box that forwards typing into the live session (alternative to typing in the session directly)
- During failure: agent's suggested next steps + Retry button
- When done: a link to the PR + a link to view `summary.md`

### View 2 — New Workflow Form

- Project picker (defaults to focused project)
- Big text area for description
- Optional Title field (auto-fills from first line if blank)
- Branch field with "auto" option or typed name; warns inline if name is in use on this project
- Start button

## Resume on Restart

If CliDeck is closed mid-workflow and reopened, the plugin scans `~/.clideck/plugins/<plugin-id>/workflows/` for any `state.json` whose `currentStage` is not `done` or `failed`. Each becomes a row in the panel with a "Resume" button. Resuming re-spawns the session for the current stage with starter-prompt context built from `state.json` and any partial markers in `done/`.

## Final Summary Document

At the end of every workflow (success or terminal failure), the orchestrator writes `summary.md` in plain non-developer language covering:

- What the user asked for
- What was built (in human terms, not file diffs)
- Manual setup required (linked to the Rhythm task)
- What the smoketest verified
- Anything still broken or skipped, with suggested next steps

`summary.md` is committed to the branch, used as the PR description, and rendered in the panel when the user clicks the finished workflow.

## CliDeck Plugin Surface

The plugin uses the standard CliDeck plugin API (`/Users/ajhochhalter/Documents/clideck/plugin-loader.js`):

- **Manifest** (`clideck-plugin.json`): id, version, install: `npm`, settings (e.g., default branch-name strategy, Rhythm MCP toggle, max fix attempts).
- **Backend** (`index.js`): registers `addToolbarAction`, `onFrontendMessage` for panel actions, `addSessionPill` for per-workflow progress, `createSession` / `closeSession` for stage sessions, `onShutdown` for cleanup. Watches workflow `done/` folders via `fs.watch` to advance stages. Stage transitions are gated by marker files; the smoketest lock is a single in-process mutex.
- **Frontend** (`client.js`): toolbar button, two-view panel rendered into a plugin-provided container, WebSocket messages over `plugin.<id>.<event>`.
- **State directory:** `~/.clideck/plugins/<plugin-id>/workflows/<workflow-id>/` (per spec above).

## Open Items to Verify Before Implementation

1. **Rhythm App MCP availability** — confirm the Rhythm MCP server is configured and reachable from a Claude session running inside CliDeck. Plugin init does a one-shot probe; if it fails, banner is shown and Stage 3 falls back to "manual list only."
2. **Codebase exploration skill** — confirm which exact skill name Opus should invoke for thorough Haiku-driven exploration (the user has both Explore and feature-dev:code-explorer available).
3. **Stage-completion signaling via marker file** — confirmed as the handoff mechanism (file watch in `done/`), not terminal-output sentinels.
4. **CI watching from sub-agents** — confirm the GitHub CLI (`gh`) is available and authenticated in the project's working directory; sub-agents need it for `gh pr checks` polling.

## Non-Goals

- Editing or replacing the existing `dev-planner` skill (this plugin replaces it for the workflow's purposes by combining steps 1+2 of dev-planner into Stages 1+2 of this plugin).
- Cross-project workflows (one workflow = one project).
- Parallel issue execution within a single workflow (deliberately sequential per Stage 3 design).
- Authoring or hosting the Rhythm App MCP server itself; the plugin only consumes it.
